Chapter II

 

           

1.      UDC and UDC Type

 

Purpose:

 

The values of the user defined code indicate the valid set of data that is specific to your business for a particular field. Each user defined code is either hard-coded or soft-coded. You can customize fields in your system by setting up user defined codes to meet the needs of your business environment. User defined codes are stored in tables that relate to a specific system and code type.

 

When you enter a value in a user defined code field the application checks the associated table to make sure it is a valid value. If no match is found, the program returns an error message “Type Mismatch”.

A)        Adding Or Changing a User defined Code Type

 

A code type categorizes your user defined code values within a product code. When you add user defined code values, you are adding them to a specific code type; and therefore the code type must already exist before adding any values to it.

 

On the system Administration Menu

 

1.       Choose a code type to change or click Add to create a new type. Note (The code with Flag=1 means you

                  cannot change this UDC as this has been defined for a particular company).

2.       To add a new code type, in the last row of the grid, complete the following fields.

a)                  Product Code (can give any number to your UDC)

b)                  UDC Type (Should be max of  max 5 characters)

c)                   Text (Description)

d)                  Company (company that UDC belongs to, or you can leave it blank which means it

                        belongs to all)

e)                  Flag (If flag=0 means you can update this UDC, if flag=1 means you cannot modify it)

 

3.       You can change or add multiple code types during the same session by pressing Enter, Click OK buttons

      When you have finished.

Chapter VI

 

1.      How to provide security to a Role

Purpose:

            The Purpose of this screen is to give and take away the access of different modules, screens, fields and

buttons to different user/role of different companies. 

 

 

 

This screen displays the list of all the Role Names with their respective unique ID’s. After we have entered the information in the Address Book and created the User Profile, next task is to assign the Role Security. In this we can enable and disable or hide, the options that we don’t want the new user to see and change. This is for the security purposes.

 

When we have several user’s, we don’t want all the users to have all the access rights, so from the Role Security, we can give the selected rights to the users.

 

For Example, Admin has all the rights to View and make changes anywhere. Where as, we don’t want all the

users to have all rights. So we can restrict the rights depending upon what kind of Role is assigned to the user.

 

From Role, we can select the access level at the Menu, screen and controls level.

 

A)        At Menu Level

 

From here, we can VIEW or HIDE, the screens that we don’t want the user to access. For Example, if we Hide the Support and Change Menu’s, then under that particular user’s Login, he wont see these menu’s.

So he won’t be able to view data and information under these Menu’s and further, he wont be able to make any changes.

 

 

 

B)        Screen Level

 

Then we can select a particular Menu, and further we can apply security at the Screen Level.

 

 

 

From here, we can again, View and Hide, the particular screens, that we don’t want the user to view and make changes. If we select View, then user will be able to view all the information and data under that particular screen. He can navigate through the records.

 

If we select Hide, then he won’t be able to view the screen. So he can’t even see the information and data in that particular screen.

 

Further, we can also hide, Disable and Enable the buttons on these screens. Select a screen, and then in the Controls we can select the rights that we want to give.

 

 

c)        Control Level

 

 

From here, we can Hide, View, Disable and enable the buttons according to the access level that we want to provide to the user.  

 

If we select Hide, user won’t be able to see that button. If we select Disabled, the button will be disabled, i.e. no action will happen on clicking the button.

 

For Example, for Delete button, if we make it View and Disabled then, user will be able to View the button the screen but if he clicks on it nothing with happens as its disabled.

 

So, this is for the Security purpose, the user will not be able to Delete any records.
